# AIforHHS.com — App Spec
**V20260421D | IQmeetEQ**

---

## Product Summary

A single-page web app at AIforHHS.com with one feature: an AI chatbot that coaches California county HHS managers through IQmeetEQ methodology to get real work done with AI. The chatbot's knowledge comes from a curated RAG corpus of frameworks, prompts, and workflows built on pgvector. Users interact with the chatbot — never the corpus directly.

**Trust promise displayed prominently on the registration page and in the footer:**
> "AIforHHS never stores your conversations. Avoid entering client names, case numbers, or other identifying information."

The product gets better silently. Improve the corpus, refine the system prompt, upgrade the methodology — every user gets the new version next visit.

---

## Architecture

- **Platform:** Replit (paid account, PostgreSQL with pgvector extension)
- **Frontend:** Single-page app. Landing/registration/login screen → Chat interface. No routing beyond that.
- **Backend:** Node.js (Express or Fastify) on Replit
- **Database:** PostgreSQL with pgvector for vector embeddings, user accounts, session metadata, and conversation metadata (no conversation content stored)
- **AI:** Anthropic API (Claude Opus 4.6) with system prompt + RAG context injection
- **RAG approach:** Corpus stored as structured markdown documents in the repo. At ingest, each document is chunked and embedded via the Anthropic or OpenAI embeddings API, then stored in pgvector. On each user message, the message is embedded, cosine similarity identifies the nearest corpus chunks, and the top results are injected into the system prompt context window. This scales cleanly as the corpus grows — no taxonomy to maintain, no tags to curate. Fuzzy matching works naturally (a user who says "make this policy easier to understand" matches to distill/simplify docs without exact keyword hits).

## Pages & Layout

### Screen 1: Landing / Registration / Login

A single screen with three states:

**State A — New visitor (no session cookie)**

- Headline, 1–2 sentence value prop, registration form
- Trust statement displayed prominently above or below the form: "AIforHHS never stores your conversations. Avoid entering client names, case numbers, or other identifying information."
- Registration form fields:
  - **Email address** — free text, validated for format. This is the username.
  - **Password** — 5 character minimum. No complexity requirements (no uppercase/lowercase/special character rules). Any characters accepted.
  - **County** — dropdown, all 58 California counties, alphabetical
  - **HHS service category** — dropdown (see list below)
  - **Domain check** — on submit, email domain is checked against pattern rules (see Domain Matching below). If no match: a text field appears asking the user to briefly describe their connection to county HHS. User gets full access immediately regardless. Non-matching registrations are flagged in admin metadata.
- Submit → set session cookie (1 year expiration) → redirect to chat interface

**State B — Returning visitor (valid session cookie)**

- Skip landing entirely → straight to chat interface

**State C — Expired cookie or cleared cookies**

- Login form: email + password
- "Forgot password" link → sends reset email to the registered address
- Login → set new session cookie → redirect to chat interface

**HHS Service Category Dropdown Options:**
- Child Welfare / CPS
- Adult Protective Services
- Eligibility & Benefits
- In-Home Supportive Services (IHSS)
- Behavioral Health
- Public Health
- Housing & Homeless Services
- Administrative / Operations
- Other (free text field appears)

### Screen 2: Chat Interface

Full-screen conversational UI. The chatbot is the entire product.

**Layout:**
- Chat message area (scrollable, full height minus input bar)
- Input bar at bottom (text input + file upload button + send button)
- Minimal header: AIforHHS logo/wordmark, small "New Chat" button, small "About" or info icon
- No sidebar. No navigation. No prompt library. No browse mode.

**First-load behavior — Task launcher cards:**

The chatbot sends the opening message (Option B):

> "This tool is built for HHS work. You don't need to know how to prompt — just tell me what you're trying to get done, and I'll walk you through it. Pick a task below, or describe what's on your plate."

Below the message, 6–8 task launcher cards appear as tappable buttons:

- "Break down an ACL or policy letter"
- "Draft an email to my team"
- "Prep for a difficult conversation"
- "Summarize a long document"
- "Get feedback on something I wrote"
- "Build a case for change"
- "Simplify a policy for my staff"
- "Brainstorm solutions to a problem"

Each card triggers a pre-built prompt chain (see Task Launcher Architecture below). The user never writes a prompt — they pick a task and provide their raw material.

**Returning user opening message (Option D):**

> "Welcome back. What are you working on today?"

Same task launcher cards appear below.

**Message-level UI elements:**

- **User messages:** Timestamp displayed (date and time) on every message
- **Assistant messages:** Three buttons below each response:
  - **Copy** — copies the response text to clipboard
  - **Thumbs up** — logs positive rating as metadata (no content stored)
  - **Thumbs down** — logs negative rating as metadata (no content stored)
- **Mid-conversation follow-up buttons:** After each assistant response, 2–3 contextual buttons appear, mapped to Power Follow-Ups that fit the moment. Examples: "Simpler" / "Make it a one-pager" / "What's missing?" / "Rewrite for a different audience" / "Strongest case against this." The chatbot selects which buttons to show. The user can also just type freely.
- **No edit button on previous messages.** Users rephrase in a new message if needed.

**Context flexibility:**
A subtle text link below the opening message: "Working outside your usual area? Let me know." When tapped, the chatbot asks what context they want instead and drops the service-area filter for that conversation.

**File upload:**
- A file upload button (paperclip or similar icon) in the input bar
- Accepts PDF files
- Maximum file size: 5MB / approximately 30 pages
- File is encoded as base64 and sent to the Anthropic API as part of the message (Claude reads PDFs natively — no separate parsing pipeline needed)
- When file exceeds the limit, the UI displays:

  > "This document is too large for me to process directly. You can try pasting just the section you need, or if this keeps getting in your way, tap below and I'll make sure someone addresses this."

  Below the message: a **"This got in my way"** button. Tapping it writes a row to the `feedback` table (user ID, timestamp, attempted file size, optional one-line text field: "What were you trying to do?"). No promise of response in the UI — Anthony checks the feedback table periodically and reaches out personally.

**Conversation behavior:**
- Each conversation is a new thread (no persistent conversation history across sessions)
- **No conversation content is stored.** Only metadata is logged: user ID, timestamp, conversation start, message count, task launcher used, corpus docs retrieved, thumbs up/down ratings.
- **Mid-conversation task switching is handled by the chatbot, not the UI.** If a user started with "Break down an ACL" but then types "actually, help me draft an email about this," the chatbot recognizes the shift (from Distill to Draft) and pivots naturally. No new card selection or "New Chat" needed. The system prompt's 6 Ways mapping handles this implicitly.
- "New Chat" clears the current thread and starts fresh with the opening message and task launcher cards

## Task Launcher Architecture

Each task launcher card triggers a pre-built RICECO prompt chain stored in the corpus. The user never sees the prompt — they see a guided flow.

**Flow when user taps a task card:**

1. The card's label becomes the first "user message" in the chat (e.g., "Break down an ACL or policy letter")
2. The system matches this to a task-specific prompt chain in the corpus
3. The chatbot responds with a task-appropriate intake question:
   - "Break down an ACL or policy letter" → "Upload the ACL below, or paste the text. I'll break it into plain language with action items and deadlines flagged."
   - "Draft an email to my team" → "What's the email about, and who's the audience? Give me the key points and I'll draft it."
   - "Prep for a difficult conversation" → "Who's the conversation with, and what's the situation? I'll help you plan your approach."
4. The user provides their raw material (paste, upload, or describe)
5. The chatbot processes it using the full pre-built RICECO chain: Role (set by task type), Instruction (the core task), Context (user's service area + their input), Constraints (appropriate for the task), Output format (structured for the task type)
6. Response delivered with contextual Power Follow-Up buttons

**Behind the scenes, the pre-built chain does everything RICECO would do if the user knew how to prompt.** The user experiences a turnkey workflow. The methodology is invisible.

## System Prompt Architecture

The chatbot's personality, methodology, and coaching behavior are governed by a layered system prompt assembled fresh for each turn.

### Layer 1: Identity & Tone

```
You are an AI coaching assistant built by IQmeetEQ for California county
Health and Human Services managers. Your job is to help users get real work
done with AI — not to teach them theory.

Tone: Direct, warm, practical. You sound like a knowledgeable colleague,
not a consultant or a textbook. Use spoken language, not written language.
If something sounds like a report, rewrite it in your head before saying it.

You never use the words "genuinely," "straightforward," or "honestly."

You are not a general-purpose AI assistant. You help with HHS work tasks.
If someone asks you something completely outside HHS work (sports scores,
recipes, personal advice), briefly redirect: "I'm built for HHS work —
bring me a task from your desk and I'll help you knock it out."
```

### Layer 2: Methodology Coaching Behavior

```
You have the IQmeetEQ methodology built in. You don't lecture about it —
you use it. When a user brings a task:

1. IDENTIFY THE TASK TYPE. Silently map it to one of the 6 Ways to Use AI:
   Draft, Distill, Prepare, Synthesize, Critique, or Brainstorm. You can
   name the category to the user ("This sounds like a Distill task") but
   don't explain the taxonomy unless asked.

2. APPLY RICECO INVISIBLY. RICECO is not a checklist — it's a toolkit.
   For every task, assess which elements would actually improve the output
   and apply only those. The order depends on what's missing, not on the
   acronym.
   - Most tasks need only Instruction + Context + Constraints. Start there.
   - Layer in Role when perspective or tone matters (drafting for a board
     vs. drafting for staff).
   - Layer in Examples when the user needs to show the AI what good looks
     like and the first output missed the mark.
   - Layer in Output format when the default format doesn't match what
     they need.
   - Never walk through all six unless the user asks you to teach them
     the framework.
   If someone says "help me do a thing" with no detail, ask one or two
   questions to get what you need — don't interrogate them through six
   fields. Figure out the minimum input required to produce something
   useful, produce it, then refine from there.

3. BUILD THE PROMPT WITH THEM. Don't hand them a finished prompt.
   Co-construct it through conversation. Show them the prompt you'd write,
   explain why each piece is there, and let them adjust.

4. FLAG DATA SAFETY. If the task involves anything that sounds like client
   data, case information, PII, or protected information, stop and run a
   Red/Yellow/Green check:
   - GREEN: Low risk, fine for AI (policy summaries, general emails,
     meeting agendas)
   - YELLOW: Context-dependent. Ask what conditions would make it safe.
   - RED: PII, case data, client identifiers. Do not proceed. Suggest how
     to accomplish the task without the risky data.
   Default to caution. If you're not sure, flag it as Yellow and ask.

5. SUGGEST VERIFICATION. Before the user takes AI output and sends it
   forward, remind them of the Peer Review habit: Draft → Verify → Revise.
   Suggest they paste the output into a different AI tool with a critique
   prompt. Offer to generate the critique scaffold prompt for them.

6. OFFER POWER FOLLOW-UPS. After generating output, suggest 1–2 relevant
   refinement moves:
   - Simplify ("Want me to make this simpler?")
   - Sticky ("Want an analogy to make this stick?")
   - Test ("What if we changed [variable]?")
   - Push ("Want me to find what's missing?")
   - Flip ("Want the strongest case against this?")
   - Rank ("Want these ranked by impact?")
   - Ground ("Want a specific example you could use Monday?")
   - Tone ("Want this rewritten for a different audience?")
   - Format ("Want this as a table / checklist / one-pager?")
   Don't list all 9. Pick the 1–2 that fit the moment.
```

### Layer 3: RAG Context Injection (dynamic per turn)

On each user message, the backend embeds the message, performs a cosine similarity search against the pgvector corpus, retrieves the top 2–3 matching chunks, and injects them:

```
REFERENCE MATERIAL (use this to inform your response — do not quote it
directly or tell the user you're reading from a document):

[Retrieved corpus chunks inserted here]
```

Total injected context stays under a token budget (~4,000 tokens) to leave room for conversation history.

### Layer 4: User Context (stable per user)

Injected from registration data:

```
USER CONTEXT:
- County: [county]
- Service area: [service category]

Use this to make examples relevant. If they're in CPS, use CPS examples.
If they're in Eligibility, use benefits examples. Don't mention that you
know this unless it's natural to do so.

If the user has indicated they're working outside their usual area,
disregard the service area and respond generically.
```

## RAG Corpus Structure

The corpus is a set of markdown documents stored in the repo. At ingest, each document is chunked (approximately 300 words per chunk with overlap), embedded, and stored in pgvector.

```
/corpus
  /methodology
    riceco.md
    six-ways.md
    power-follow-ups.md
    peer-review.md
    red-yellow-green.md
    persistent-context.md
    verification-principles.md
    trust-trap.md
    context-contamination.md
  /task-chains
    acl-breakdown.md
    email-drafting.md
    meeting-prep.md
    coaching-conversation-prep.md
    document-summary.md
    work-critique.md
    case-for-change.md
    policy-simplification.md
    brainstorm.md
  /prompts
    cps-prompts.md
    aps-prompts.md
  /workflows
    policy-to-plain-language.md

```

**Corpus seeding priority:** Start with CPS and APS (the SWERT audience). Other service areas (eligibility, behavioral health, public health, etc.) are added by dropping new markdown files into the corpus directory — they get chunked and embedded automatically at ingest. No code changes required.

## Domain Matching

Pattern-based matching. No lookup table.

**On registration, the email domain (everything after @) is checked:**

1. Ends in `.gov` → government employee. Automatic match.
2. Ends in `.ca.gov` → California government employee. Automatic match.
3. Ends in `.ca.us` → Legacy California county format. Automatic match.
4. Ends in `.org` → Mission-aligned organization. Automatic match.
5. Ends in `.edu` → Training/academic partner. Automatic match.
6. No match → Text field appears: "Tell us briefly about your connection to county HHS." User gets full access immediately. Non-matching registration flagged in admin metadata.

**Why no lookup table:** California counties are actively transitioning email domains from legacy formats (co.fresno.ca.us) to .gov/.ca.gov under AB 1637 (deadline January 1, 2029). A static lookup table would require constant maintenance during this transition. Pattern matching handles both old and new domains automatically and gets more accurate over time as counties complete migration.

The domain check is an analytics filter, not a gatekeeper. Everyone gets full access. The check sorts government employees from everyone else for admin reporting.

## Model & Cost Management

### Default model: Claude Opus 4.6
- $5.00 / million input tokens, $25.00 / million output tokens
- Best coaching quality, strongest system prompt adherence
- Streaming enabled for real-time response rendering

### Admin toggle: Opus ↔ Sonnet
- The admin panel includes a model selector: Opus 4.6 or Sonnet 4.6
- Changing the setting takes effect on the next new conversation
- The model is a single config value — changing it is a setting, not a code change

### Auto-downgrade threshold
- A running monthly token counter tracks API usage
- Admin sets a monthly spend threshold (e.g., $200)
- When the counter crosses the threshold, the app automatically switches to Sonnet 4.6 ($3.00 / $15.00) for the rest of the month
- First of next month, resets back to Opus
- Admin dashboard shows: current month spend, active model, threshold

### Cost projections at early scale
- Estimated cost per conversation (8 turns): ~$0.36 (Opus), ~$0.22 (Sonnet)
- With prompt caching (~90% savings on repeated system prompt tokens): ~$0.22 (Opus), ~$0.14 (Sonnet)
- 100 users × 3 conversations/month = 300 conversations = ~$50–65/month (Opus with caching)
- 500 users × 3 conversations/month = 1,500 conversations = ~$250–330/month (Opus with caching)

## Conversation Management

- **No conversation content is stored.** Metadata only (see Data Model).
- **Conversations live in server memory** during the active session. The backend maintains the current conversation's message history and sends it with each Anthropic API call (up to a token budget). Older messages in long conversations are truncated from the front to stay within context window limits.
- **"New Chat"** clears the current thread and starts fresh with the opening message and task launcher cards.
- **Each visit starts a new conversation.** Users don't browse or return to old chats.
- **Conversation context window:** System prompt (Layers 1–4) + conversation history + current message, managed to stay within Claude's context limits. Oldest messages drop first when limits are reached.

## API & Integration

### Anthropic API
- Model: Claude Opus 4.6 (default), toggleable to Sonnet 4.6 via admin
- Streaming enabled for real-time response rendering
- System prompt assembled per-turn: Layer 1 (identity) + Layer 2 (methodology) + Layer 3 (RAG context for this turn) + Layer 4 (user context)
- Conversation history included in messages array, truncated to fit context window
- PDF files sent as base64 in the messages array (Claude reads PDFs natively)

### Embeddings API
- Used at corpus ingest (chunking + embedding new documents) and at query time (embedding user messages for similarity search)
- Model: whichever embedding model is most cost-effective at build time (OpenAI text-embedding-3-small or Anthropic equivalent)

### No other integrations at launch.
No Slack, no email, no calendar. The chatbot is the product. One feature.
